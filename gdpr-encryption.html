Fundamentals

What is Encryption?

Encryption transforms readable data (plaintext) into an unreadable format (ciphertext) using a mathematical algorithm and a key. The original value can be recovered — but only by someone who holds the key.

Symmetric

One Key for Both Directions

  • Same key encrypts and decrypts
  • Fast — suitable for large data volumes
  • Example: AES-256 (Delta Lake at-rest)
  • Risk: one leaked key exposes everything
Asymmetric

Public Encrypts, Private Decrypts

  • Two mathematically linked keys
  • Anyone can encrypt with the public key
  • Only the private key holder can decrypt
  • Example: RSA, used in TLS / data sharing

🔑 Encryption is reversible — the original value can always be recovered with the right key. This makes it fundamentally different from pseudonymization.

Fundamentals — Getting the Terms Right

Encryption vs Pseudonymization

Reversible

Encryption

  • Original value recoverable with the key
  • Leak the key → all data exposed at once
  • Good for: data in transit, at-rest storage, fields you display back to users
  • Under GDPR: still fully personal data
What We Use

Pseudonymization (HMAC)

  • Original value is not recoverable
  • Leak all salts → still cannot reverse the hash
  • Deterministic: same input + salt = same hash
  • Under GDPR: Article 25 rewards it with reduced obligations

📌 We use pseudonymization for PII in the Lakehouse. A full breach of Delta tables yields nothing without the source plaintext.

GDPR Article 25 — Pseudonymization

Why Pseudonymization Is the Right Choice

What it means legally

  • Personal data processed so it can no longer be attributed to a specific person without additional information
  • Salt store must be kept separately and securely
  • Data is still personal data — GDPR still applies
  • Article 25 treats it as a gold-standard technical control

What you gain

  • Reduced breach notification obligation (Recital 83)
  • Supports Privacy by Design (Article 25)
  • Crypto-erasure satisfies Right to Erasure (Article 17)
  • Breach of Lakehouse ≠ notifiable breach if salt store is separate

🔑 The salt store is the legal boundary. Keep it apart from the Lakehouse — that separation is your compliance posture.

The Algorithm

What is HMAC?

Hash-based Message Authentication Code

How it works

  • Takes a value + a secret salt
  • Produces a fixed-length hash
  • Same inputs → always the same output
  • Different salt → completely different hash

The security guarantee

  • Given hash + salt → cannot recover the original
  • Even if all salts leak, original values are safe
  • A leaked database without source data is useless
  • Joins still work — the hash is a stable fingerprint

🔑 HMAC("123-45-6789", salt_jitesh) → "a9f3c7d2..." — that arrow only ever points one way.

GDPR Architecture

Per-User Salts

Each user gets their own unique salt, generated once and stored securely. Even if every salt is exposed, no attacker can reverse the hash back to the original PII.

🔑 Salt Store

  • user_id → unique random salt
  • Stored in secrets manager or protected Delta table
  • Never co-located with the encrypted data

Isolation by design

  • One leaked salt only affects one user
  • No single point of cryptographic failure
  • Breach of the Lakehouse ≠ breach of PII
  • Salt deletion = cryptographic erasure (GDPR Art. 17)

Implementation

Row-by-Row Encryption at Ingest

# At Bronze ingest — one pattern, any table for row in incoming_stream: user_id = row["user_id"] user_salt = salt_store.get(user_id) # lookup from secure store for col in pii_columns: # ["ssn", "email", "device_id", ...] row[col] = hmac_sha256(row[col], user_salt) bronze_table.write(row) # only hashed values land in Delta

📌 HMAC is table-agnostic — only the pii_columns config changes per table. The function and salt store are universal across the entire Lakehouse.

GDPR Right to Erasure — The Challenge

You Can't Reverse HMAC to Find Related Records

Jitesh submits a GDPR deletion request. You know who Jitesh is. You know his salt.
But you cannot recover his device IDs, emails, or phone numbers — because HMAC is one-way.

Problem

What you can't do

  • Given hmac(device_id, salt), you cannot get device_id back
  • You can't query dim_devices WHERE device_id = ?
  • You don't know which rows across which tables belong to Jitesh
Solution

Store a mapping of hashed identities

  • Build a record linking Jitesh's hashed user ID to all his other hashed entity IDs
  • Must be built at ingest time, before you lose the plaintext
  • The mapping itself stores only hashed values — no plaintext PII
  • This is called the Anchor Table
Design Options

Three Approaches to GDPR Deletion Mapping

Option 1

Central PII Store

  • Store every raw PII value per user in one table
  • ✅ Simple to reason about
  • ❌ Massive single point of failure
Option 2

Per-Table Mapping

  • Each table tracks its own user → row mapping
  • ✅ Blast radius limited per table
  • ❌ Easy to miss a table
Option 3 ✅ Recommended

Anchor Table

  • One table of hashed IDs only — no plaintext
  • Single lookup enables full deletion
  • ✅ No PII stored; minimal exposure
